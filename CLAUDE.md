# CLAUDE.md

このファイルは、このリポジトリで作業する際にClaude Code (claude.ai/code) に提供されるガイダンスです。

## 設計思想

### ファイル構成の原則
- **機能ごとにファイルを分割** - Go言語の思想に従い、各ファイルは単一の責務を持つ
- 1ファイル1機能を基本とし、関連する機能のみをグループ化する
- ファイル名は機能を明確に表現する

### 開発の3原則

1. **KISS原則 (Keep It Simple, Stupid)**
   - シンプルな解決策を優先する
   - 複雑さは必要最小限に抑える
   - 読みやすさを最優先する

2. **YAGNI原則 (You Aren't Gonna Need It)**
   - 今必要な機能だけを実装する
   - 将来必要になるかもしれない機能は実装しない
   - 過剰な抽象化を避ける

3. **DRY原則 (Don't Repeat Yourself)**
   - 同じコードを繰り返さない
   - 共通処理は関数やモジュールに抽出する
   - ただし、無理な共通化よりも明確さを優先する

## コミットルール

### 基本ルール

1. **絵文字プレフィックス**
   - ✨ `:sparkles:` - 新機能
   - 🐛 `:bug:` - バグ修正
   - 📝 `:memo:` - ドキュメント
   - ♻️ `:recycle:` - リファクタリング
   - 🎨 `:art:` - コードの構造/フォーマット改善
   - ⚡ `:zap:` - パフォーマンス改善
   - 🔧 `:wrench:` - 設定ファイル変更
   - ✅ `:white_check_mark:` - テスト追加/更新
   - 🚀 `:rocket:` - デプロイ関連
   - 🚮 `:put_litter_in_its_place:` - コード/ファイル削除

2. **日本語で記述**
   - コミットメッセージは日本語で書く
   - 簡潔かつ明確に変更内容を説明する

3. **アトミックコミット**
   - 1つのコミットは1つの論理的な変更のみを含む
   - 複数の異なる変更は別々のコミットに分ける
   - 各コミットは独立してビルド可能な状態を保つ

### コミットメッセージ例

```
✨ ユーザー認証機能を追加

🐛 ログイン時のセッションタイムアウトを修正

📝 APIドキュメントを更新

🚮 未使用のヘルパー関数を削除
```
